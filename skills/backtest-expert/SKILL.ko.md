# 백테스트 전문가

낙관적인 결과보다 강건성을 우선하는 전문 방법론에 기반해 트레이딩 전략을 백테스팅하는 체계적 접근입니다.

## 핵심 철학

**목표**: 종이 위에서 "가장 많이 수익 나는" 전략이 아니라, "가장 덜 망가지는" 전략을 찾는다.

**원칙**: 마찰을 추가하고, 가정을 스트레스 테스트하고, 무엇이 살아남는지 확인한다. 비관적인 조건에서도 전략이 유지되면 실전 트레이딩에서도 작동할 가능성이 높다.

## 이 스킬을 사용할 때

다음 상황에서 이 스킬을 사용합니다:
- 체계적 트레이딩 전략을 개발하거나 검증할 때
- 트레이딩 아이디어가 실전 적용에 충분히 강건한지 평가할 때
- 백테스트가 왜 오해를 부를 수 있는지 원인을 찾을 때
- 올바른 백테스팅 방법론을 학습할 때
- 일반적인 함정(curve-fitting, look-ahead bias, survivorship bias)을 피할 때
- 파라미터 민감도와 시장 국면 의존성을 평가할 때
- slippage 및 실행 비용에 대한 현실적인 기대치를 설정할 때

## 사전 요구사항

- Python 3.9+ (평가 스크립트용)
- API key 불필요
- 외부 데이터 의존성 없음 — 메트릭은 사용자가 제공

## 워크플로

### 1. 가설 명시

엣지를 한 문장으로 정의합니다.

**예시**: "실적 발표 후 3% 이상 갭 상승한 주식이 첫 1시간 내 전일 종가까지 되돌리면 mean-reversion 기회가 발생한다."

엣지를 명확히 설명할 수 없다면 테스트를 진행하지 마세요.

### 2. 재량 없는 규칙으로 코딩

아래를 완전한 구체성으로 정의합니다:
- **진입**: 정확한 조건, 타이밍, 가격 유형
- **청산**: stop loss, profit target, 시간 기반 청산
- **포지션 사이징**: 고정 $$, 포트폴리오 비율 %, 변동성 조정
- **필터**: 시가총액, 거래량, 섹터, 변동성 조건
- **유니버스**: 어떤 종목이 대상인지

**중요**: 주관적 판단은 허용하지 않습니다. 모든 의사결정은 규칙 기반이며 모호함이 없어야 합니다.

### 3. 초기 백테스트 실행

다음 조건으로 테스트합니다:
- **최소 5년** (가능하면 10년 이상)
- **여러 시장 국면** (상승장, 하락장, 고/저 변동성)
- **현실적 비용**: 수수료 + 보수적인 slippage

초기 결과를 보고 기본적인 실행 가능성을 점검합니다. 구조적으로 망가진 전략이면 가설을 수정해 반복합니다.

### 4. 전략 스트레스 테스트

테스트 시간의 80%를 여기에 써야 합니다.

**파라미터 민감도**:
- stop loss를 기준값의 50%, 75%, 100%, 125%, 150%로 테스트
- profit target을 기준값의 80%, 90%, 100%, 110%, 120%로 테스트
- 진입/청산 타이밍을 ±15-30분 변동
- 좁은 스파이크가 아닌, 안정적인 성과 "plateau"를 찾기

**실행 마찰**:
- slippage를 일반 추정치의 1.5-2배로 증가
- 최악 체결 가정 모델링(매수 ask+1 tick, 매도 bid-1 tick)
- 현실적인 주문 거절 시나리오 추가
- 비관적인 수수료 구조로 테스트

**시간 강건성**:
- 연도별 성과 분석
- 과반수 연도에서 양(+)의 expectancy 요구
- 1-2개 예외적 구간에 의존하지 않는지 확인
- 시장 국면별로 분리 테스트

**표본 크기**:
- 절대 최소: 30회 거래
- 권장: 100회 이상 거래
- 높은 신뢰도: 200회 이상 거래

### 5. Out-of-Sample 검증

**Walk-forward analysis**:
1. 학습 구간(예: 1-3년)에서 최적화
2. 검증 구간(4년차)에서 테스트
3. 롤링하며 반복
4. in-sample vs out-of-sample 성과 비교

**경고 신호**:
- out-of-sample 성과가 in-sample의 50% 미만
- 파라미터 재최적화가 자주 필요
- 기간마다 파라미터가 크게 변함

### 6. 결과 평가

**답해야 할 질문**:
- 엣지가 비관적 가정에서도 살아남는가?
- 파라미터 변화 전반에서 성과가 안정적인가?
- 여러 시장 국면에서 작동하는가?
- 통계적 신뢰를 위한 표본 크기가 충분한가?
- 결과가 "너무 좋아서 의심스러운" 수준은 아닌가?

**의사결정 기준**:
- ✅ **Deploy**: 모든 스트레스 테스트를 통과하고 성과가 수용 가능
- 🔄 **Refine**: 핵심 로직은 타당하지만 파라미터 조정 필요
- ❌ **Abandon**: 스트레스 테스트 실패 또는 취약한 가정에 의존

구조화된 정량 평가를 위해 평가 스크립트를 사용하세요:

```bash
python3 skills/backtest-expert/scripts/evaluate_backtest.py \
  --total-trades 150 \
  --win-rate 62 \
  --avg-win-pct 1.8 \
  --avg-loss-pct 1.2 \
  --max-drawdown-pct 15 \
  --years-tested 8 \
  --num-parameters 3 \
  --slippage-tested \
  --output-dir reports/
```

이 스크립트는 5개 차원(Sample Size, Expectancy, Risk Management, Robustness, Execution Realism)을 점수화하고, red flag를 감지해 Deploy/Refine/Abandon 판정을 출력합니다.

## 핵심 테스트 원칙

### 전략을 가혹하게 검증하라

모든 곳에 마찰을 추가합니다:
- 실제보다 높은 수수료
- 일반 대비 1.5-2배 slippage
- 최악 체결
- 주문 거절
- 부분 체결

**근거**: 비관적 가정을 견디는 전략은 실전에서 백테스트 대비 더 잘 작동하는 경우가 많습니다.

### 피크가 아니라 Plateau를 찾아라

성과 스파이크를 만드는 최적값이 아니라, 성과가 안정적인 파라미터 구간을 찾습니다.

**좋은 예**: stop loss가 1.5%~3.0% 어디여도 수익
**나쁜 예**: stop loss가 정확히 2.13%일 때만 작동

안정적 성과는 진짜 엣지를 시사하고, 좁은 최적점은 curve-fitting 가능성을 시사합니다.

### 선별 사례가 아니라 전체 사례를 테스트하라

**잘못된 접근**: 잘됐던 "시장 리더" 사례만 손으로 고름
**올바른 접근**: 기준을 충족한 모든 종목(실패 사례 포함)을 테스트

선별 사례는 survivorship bias를 만들고 전략 품질을 과대평가합니다.

### 아이디어 생성과 검증을 분리하라

**직관**: 가설 생성에는 유용
**검증**: 반드시 데이터 기반으로만 수행

아이디어에 대한 애착이 테스트 결과 해석에 영향을 주지 않게 하세요.

## 흔한 실패 패턴

시간을 아끼려면 다음 패턴을 조기에 인식하세요:

1. **파라미터 민감도**: 특정 파라미터 값에서만 작동
2. **국면 의존성**: 어떤 해에는 훌륭하지만 다른 해에는 참담
3. **Slippage 민감도**: 현실적 비용 반영 시 비수익
4. **작은 표본**: 통계 신뢰를 얻기엔 거래 수 부족
5. **Look-ahead bias**: "너무 좋아서 의심스러운" 결과
6. **과최적화**: 파라미터는 많고 out-of-sample은 나쁨

자세한 예시와 진단 프레임워크는 `references/failed_tests.md`를 참고하세요.

## 중요 리마인더

**시간 배분**: 아이디어 생성 20%, 아이디어 깨뜨리기 80%.

**Context-free 요구사항**: 전략이 "완벽한 맥락"에서만 작동한다면 체계적 트레이딩에 충분히 강건하지 않습니다.

**Red flag**: 백테스트 결과가 너무 좋게 보이면(승률 90%+, 미미한 drawdown, 완벽한 타이밍), look-ahead bias나 데이터 이슈를 면밀히 점검하세요.

**도구 한계**: 사용하는 백테스팅 플랫폼의 특성(보간 방식, 저유동성 처리, 데이터 정렬 문제)을 이해하세요.

**통계적 유의성**: 작은 엣지를 입증하려면 큰 표본이 필요합니다. 거래당 5% 엣지는 운과 구분하려면 100회 이상 거래가 필요합니다.

## 재량형 vs 체계형 차이

이 스킬은 다음 조건의 **체계적/정량적** 백테스팅에 초점을 둡니다:
- 모든 규칙을 사전에 코딩
- 실행 중 재량이나 "감" 배제
- 선별 사례가 아닌 전체 과거 사례 테스트
- 맥락(뉴스, 매크로)을 의도적으로 제거

재량 트레이더의 학습 방식은 다릅니다. 주관적 판단이 필요한 셋업에는 이 스킬이 맞지 않을 수 있습니다.
