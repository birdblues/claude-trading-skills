# 실패한 백테스트에서 배우기

## 목차

1. 실패한 아이디어가 가치 있는 이유
2. 흔한 실패 패턴
3. 케이스 스터디 프레임워크
4. Red Flags 체크리스트

## 1. 실패한 아이디어가 가치 있는 이유

### 실패의 가치

**핵심 인사이트**:
- 실패한 테스트는 실전 적용을 막아 자본을 지켜준다
- 실패 패턴은 어떤 가정이 성립하지 않는지 보여준다
- 작동하지 않는 것을 이해하면 탐색 공간이 좁아진다
- 실패한 테스트는 취약한 전략을 식별하는 경험을 쌓게 해준다

### 문서화 규율

**실패한 아이디어마다 기록할 항목**:
- 테스트한 가설
- 왜 작동할 것이라 생각했는지
- 데이터가 무엇을 보여줬는지
- 구체적인 파손 지점
- 배운 교훈

**목적**: 같은 실수를 반복하지 않도록 "anti-pattern" 라이브러리를 축적한다.

## 2. 흔한 실패 패턴

### 패턴 1: 파라미터 민감도

**증상**: 전략이 매우 특정한 파라미터 값에서만 작동.

**예시 시나리오**:
- stop loss가 정확히 2.5%일 때만 수익
- 3%로 올리거나 2%로 내리면 성과가 크게 악화
- 안정적 성과의 "plateau"가 없음

**실패 이유**: 실제 시장에는 노이즈가 존재하므로, 작은 변화에 전략이 망가지면 신호가 아니라 노이즈를 포착했을 가능성이 높다.

**교훈**: 파라미터 범위 전반에서 성과가 안정적인 전략을 찾아야 한다.

### 패턴 2: 국면 의존 성과

**증상**: 어떤 해에는 매우 잘 작동하고, 다른 해에는 매우 나쁨.

**예시 시나리오**:
- 2017-2019년(저변동성 상승장)에는 우수한 성과
- 2020년(고변동성)에는 파국적 손실
- 2022년(하락 추세)에는 저조한 성과

**실패 이유**: 특정 시장 조건에 의존하며, 다양한 환경에서 버틸 만큼 강건하지 않다.

**교훈**: 모든 국면에서 최고일 필요는 없지만, 수용 가능한 성과를 요구해야 한다.

### 패턴 3: Slippage 민감도

**증상**: 현실적인 거래 비용을 추가하면 전략이 비수익으로 변함.

**예시 시나리오**:
- 백테스트에서 거래당 평균 수익 0.5%
- 편도 0.1% slippage(왕복 0.2%) 추가 시 수익 소멸
- 비현실적인 체결을 가정해야만 수익 유지

**실패 이유**: 실전 마찰을 버틸 만큼 엣지가 충분히 크지 않다.

**교훈**: 비용에 대한 비관적 가정을 견딜 수 있을 만큼 엣지가 커야 한다.

### 패턴 4: 표본 크기 문제

**증상**: 적은 거래 수에 기반한 강한 성과.

**예시 시나리오**:
- 백테스트 승률 80%
- 5년 동안 총 거래 15회뿐
- 결과가 몇 건만 달라도 성과가 크게 바뀜

**실패 이유**: 운과 엣지를 구분할 만큼 데이터가 충분하지 않다.

**교훈**: 의미 있는 결론을 위해 최소 100회 거래, 가능하면 200회 이상을 요구해야 한다.

### 패턴 5: Look-Ahead Bias

**증상**: 완벽하거나 거의 완벽한 백테스트 결과.

**예시 시나리오**:
- 전략 승률 95%+
- 비현실적으로 좋은 진입/청산 타이밍
- 현실적으로 보기 어려울 정도로 우수한 성과

**실패 이유**: 거래 시점에 알 수 없는 정보를 사용했을 가능성이 높다.

**교훈**: "너무 좋아서 의심스러운" 결과를 의심하고 데이터 정렬을 면밀히 감사해야 한다.

### 패턴 6: 과최적화 (Curve Fitting)

**증상**: 파라미터가 많은 복잡한 전략이 in-sample에서는 우수하지만 out-of-sample에서는 부진.

**예시 시나리오**:
- 전략에 8-10개 indicator와 특정 임계값 사용
- in-sample 성과: 연 40% 수익
- out-of-sample 성과: 연 -5% 수익
- 파라미터를 지속적으로 재최적화해야 함

**실패 이유**: 실제 시장 구조가 아닌 과거 노이즈에 맞춰졌다.

**교훈**: 파라미터가 적은 단순 전략을 선호하고, 강한 out-of-sample 결과를 요구해야 한다.

## 3. 케이스 스터디 프레임워크

### 실패한 아이디어 문서화 템플릿

백테스트가 실패했을 때 이 프레임워크를 사용하세요:

#### 1. 초기 가설
- **어떤 엣지를 포착하려 했는가?**
- **왜 이것이 작동할 것이라 생각했는가?**
- **논리적 근거는 무엇이었는가?**

#### 2. 구현 세부사항
- **진입 규칙** (구체적이고 완전하게)
- **청산 규칙** (stop loss, profit target, 시간 기반)
- **포지션 사이징**
- **필터 또는 조건**

#### 3. 테스트 결과
- **기본 메트릭**:
  - 총 거래 수
  - 승률
  - 평균 수익/손실
  - Max drawdown
  - 연도별 수익률

- **파라미터 민감도**:
  - 파라미터 변형에 따른 결과 변화
  - 안정적 성과의 "plateau" 존재 여부

- **국면 분석**:
  - 서로 다른 시장 환경에서의 성과
  - 문제가 발생한 국면

#### 4. 파손 지점
- **전략이 구체적으로 무엇 때문에 실패했는가?**
  - Slippage가 너무 높았는가?
  - 파라미터 민감도가 과도했는가?
  - 특정 국면 의존인가?
  - 표본 크기가 부족한가?

#### 5. 교훈
- **어떤 가정이 틀렸는가?**
- **다음에는 무엇을 다르게 테스트할 것인가?**
- **살릴 수 있는 요소가 있는가?**

### 예시: 실패한 모멘텀 역추세 전략

#### 1. 초기 가설
강한 모멘텀 이후 mean reversion을 포착하려 했다. 가설: 실적 발표로 5%+ 갭 상승한 주식은 추세를 이어가기 전에 2-3% 되돌림이 자주 발생해 단기 역추세 기회를 제공한다.

#### 2. 구현
- 진입: 실적 발표일 시가에 5%+ 갭 상승 시 공매도 진입
- 청산: 2% 이익에서 커버 또는 3% stop loss
- 보유 기간: 최대 3일
- 필터: 시가총액 >$2B, 평균 거래량 >500K shares

#### 3. 테스트 결과
- 5년 동안 67회 거래
- 승률: 58%
- 평균 수익: 2.1%, 평균 손실: 3.2%
- Max drawdown: 18%
- 2019-2021: 수익
- 2022-2023: 큰 손실

#### 4. 파손 지점
- 강한 모멘텀 환경(2021 meme stocks)에서 전략 실패
- 지속 상승 모멘텀에서 stop loss가 빈번히 체결
- 갭 상승 후 즉시 추가 상승하는 종목에서 손실 과대
- 작은 표본(67회 거래)으로 통계 신뢰 낮음
- 고변동성 구간의 공매도 진입 slippage가 얇은 엣지를 제거

#### 5. 교훈
- mean reversion 전략은 모멘텀 국면에 취약
- 국면 필터 필요(예: 높은 VIX 또는 약세장일 때만 거래)
- 모멘텀 전략에는 5년 테스트가 부족, 10년+ 필요
- slippage를 버티기엔 엣지가 너무 작음(2% target vs 3% stop)
- 더 나은 접근: 즉시 역방향 진입이 아니라 실제 되돌림 확인 후 진입

## 4. Red Flags 체크리스트

모든 백테스트 평가 시 다음 체크리스트를 사용하세요:

### 데이터 품질 이슈
- [ ] survivorship bias를 처리했는가?
- [ ] 테스트에 상장폐지 종목이 포함되는가?
- [ ] 데이터 정렬이 올바른가(look-ahead bias 없음)?
- [ ] corporate actions(분할, 배당)을 올바르게 처리했는가?

### 표본 크기 우려
- [ ] 최소 100회 거래인가? (가능하면 200+)
- [ ] 최소 5년 데이터인가? (가능하면 10+)
- [ ] 완전한 시장 사이클을 포함하는가?
- [ ] 여러 시장 국면에서 테스트했는가?

### 파라미터 강건성
- [ ] 인접한 파라미터 값에서도 전략이 작동하는가?
- [ ] 안정적 성과의 "plateaus"가 존재하는가?
- [ ] 파라미터 수가 최소인가(이상적으로 <5)?
- [ ] 파라미터가 순수 최적화가 아니라 논리적 추론 기반인가?

### 실행 현실성
- [ ] 현실적인 수수료를 포함했는가?
- [ ] slippage를 보수적으로 모델링했는가(일반치의 1.5-2배)?
- [ ] 최악 체결을 고려했는가?
- [ ] 주문 거절/부분 체결을 반영했는가?

### 성과 특성
- [ ] 다수 연도에서 양(+)의 expectancy인가?
- [ ] 주요 모든 국면에서 수용 가능한 성과인가?
- [ ] 파국적 drawdown(>50%)이 없는가?
- [ ] 마찰을 견딜 만큼 엣지가 충분히 큰가?

### Bias 방지
- [ ] 전략을 테스트 전에 정의했는가?
- [ ] 가설에 경제적 논리가 있는가?
- [ ] 결과가 "너무 좋아서 의심스럽지" 않은가?
- [ ] out-of-sample 테스트를 수행했는가?
- [ ] 사례 cherry-picking이 없는가?

### 도구 한계
- [ ] 테스트 플랫폼의 보간 방식(interpolation methods)을 알고 있는가?
- [ ] 플랫폼의 저유동성 처리 방식을 이해하는가?
- [ ] 데이터 제공자 고유의 특성을 파악했는가?

**2-3개 이상 항목이 체크되지 않으면, 실전 적용을 고려하기 전에 백테스트 추가 작업이 필요합니다.**
