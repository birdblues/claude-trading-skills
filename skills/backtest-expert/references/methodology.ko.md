## 1. 핵심 테스트 기법

### "아이디어를 끝까지 검증" 접근

**핵심 원칙**: 종이 위에서 가장 많이 수익 나는 전략이 아니라, 가장 덜 망가지는 전략을 찾기 위해 마찰과 페널티를 추가한다.

**핵심 기법**:
- 여러 stop loss 변형
- 다양한 profit target
- 현실적 + 과장된 수수료
- 최악 체결
- 확장된 테스트 기간
- 다양한 시장 국면

### R&D 시간의 80/20 규칙

- 20%는 아이디어 생성 및 규칙화
- 80%는 스트레스 테스트와 아이디어 깨뜨리기

## 2. 스트레스 테스트 방법

### 실행 마찰 테스트

**필수 마찰 추가 항목**:
- 현실적인 수수료(실제 브로커 요율)
- 비관적 slippage(일반치의 1.5-2배)
- 최악 진입 체결(ask + 1-2 ticks)
- 최악 청산 체결(bid - 1-2 ticks)
- 주문 거절 시나리오
- 부분 체결

### 파라미터 강건성 테스트

여러 설정에서 테스트합니다:
- 진입 시점 변형(±15-30분)
- stop loss 거리(기준값의 50%, 75%, 100%, 125%, 150%)
- profit target(기준값의 80%, 90%, 100%, 110%, 120%)
- 포지션 사이징 규칙
- 필터 임계값

**목표**: 작은 파라미터 변화로 결과가 급변하지 않는 "plateau" 성능을 찾는다.

### 시간 기반 강건성

**최소 요구사항**:
- 최소 5-10년 구간 테스트
- 다음과 같은 다양한 시장 국면 포함:
  - 상승장
  - 하락장
  - 고변동성 구간
  - 저변동성 구간
  - 추세장
  - 박스권 시장

**연도별 분석**: 전략은 소수의 예외적 1-2개 연도에 의존하지 않고, 다수의 연도에서 양(+)의 expectancy를 보여야 한다.

## 3. 파라미터 민감도 분석

### Heat Map 분석

두 개 파라미터를 동시에 바꿔 2D heat map을 만듭니다:
- Profit target(행) × Stop loss(열)
- Entry time(행) × Exit time(열)
- Volatility filter(행) × Volume filter(열)

**해석**:
- 강건한 전략은 일관된 성능의 "plateau"를 보인다
- 취약한 전략은 "spike" 또는 좁은 최적 구간을 보인다
- 파라미터 경계에서 성능 절벽이 있는 전략은 피한다

### Walk-Forward 분석

1. 학습 구간(예: 1-2년)에서 파라미터 최적화
2. 해당 파라미터를 검증 구간(3년차)에 적용해 테스트
3. 롤링하며 반복
4. in-sample과 out-of-sample 성능 비교

**경고 신호**:
- out-of-sample 성능이 in-sample의 50% 미만
- 파라미터 재최적화가 빈번함
- 기간마다 파라미터가 크게 변함

### 평가 스크립트의 Profit Factor 점수

평가 스크립트는 Risk Management 차원(20점 중 0-8점)의 한 구성요소로 profit factor(PF)를 점수화합니다. 매핑은 integer truncation을 사용하는 연속 선형 보간입니다:

- PF < 1.0 → 0점 (비수익)
- PF 1.0 to 3.0 → 선형 0 to 8점: `int((PF - 1.0) / 2.0 * 8)`
- PF >= 3.0 → 8점 (상한)

`int()` truncation으로 인해 1점 단위 계단식 점수가 생깁니다(예: PF 1.25→1점, PF 1.50→2점). 이는 의도된 설계로, profit factor의 미세한 차이에 과도한 정밀도를 부여하지 않기 위함입니다.

## 4. Slippage 및 마찰 모델링

### 현실적 Slippage 가정

**시가총액 기준**:
- Mega cap (>$200B): 0.01-0.02%
- Large cap ($10B-$200B): 0.02-0.05%
- Mid cap ($2B-$10B): 0.05-0.10%
- Small cap ($300M-$2B): 0.10-0.20%
- Micro cap (<$300M): 0.20-0.50%+

**주문 유형 기준**:
- Market order: 더 높은 slippage
- Limit order: 더 낮은 slippage지만 미체결 가능성
- Stop order: 변동성 구간에서 큰 slippage 가능성

### 보수적 테스트 접근

스트레스 테스트에서는 일반 slippage 추정치의 1.5-2배를 사용합니다:
- 일반 slippage가 0.05%면 0.075-0.10%로 테스트
- 일반치가 0.10%면 0.15-0.20%로 테스트

**근거**: 비관적 가정을 견디는 전략은 실전에서 백테스트보다 더 잘 작동하는 경우가 많다.

## 5. 표본 크기 가이드라인

### 최소 거래 수 요구사항

**통계적 유의성 임계값**:
- 절대 최소: 30회 거래
- 권장 최소: 100회 거래
- 높은 신뢰도: 200회 이상 거래

**큰 표본이 중요한 이유**:
- 이상치 영향 축소
- 통계적 신뢰 확보
- 진짜 엣지와 운을 구분

### 테스트 기간 고려사항

**최소 테스트 기간**: 5년
**권장 테스트 기간**: 10년+

**반드시 포함**:
- 최소 한 번의 완전한 시장 사이클
- 여러 변동성 국면
- 다양한 Federal Reserve 정책 환경

## 6. 시장 국면 분석

### 국면 분류

**변동성 기반 국면**:
- Low volatility: VIX <15
- Normal volatility: VIX 15-25
- High volatility: VIX 25-35
- Extreme volatility: VIX >35

**추세 기반 국면**:
- Strong uptrend: 6개월 동안 시장 +10%+
- Moderate uptrend: 6개월 동안 시장 +5% to +10%
- Sideways: 6개월 동안 시장 -5% to +5%
- Downtrend: 6개월 동안 시장 <-5%

### 국면별 성과 요구사항

**강건한 전략의 특징**:
- 다수의 국면에서 양(+)의 expectancy
- 모든 국면에서 수용 가능한 성과(최고일 필요는 없음)
- 특정 단일 국면에서 파국적 실패가 없음
- 어느 국면에서 취약해지는지 이해하고 있음

## 7. 일반적인 함정과 Bias

### Survivorship Bias

**문제**: 현재 거래 중인 주식만 테스트하면 상장폐지/파산 기업이 제외된다.

**해결**: 과거 상장폐지 데이터를 포함한 survivorship-bias-free 데이터셋 사용.

### Look-Ahead Bias

**문제**: 거래 시점에 알 수 없던 정보를 사용.

**예시**:
- intraday 의사결정에 EOD 데이터 사용
- 오늘 종가 의사결정에 다음 날 시가 사용
- 미래 데이터 포인트를 포함해 indicator 계산

**예방**: 엄격한 timestamp 통제와 데이터 정렬 점검.

### Curve-Fitting (Over-Optimization)

**경고 신호**:
- 파라미터가 너무 많음 (>=7이면 over-optimization flag; <=4가 이상적, 5-6은 허용)
- 지나치게 구체적인 파라미터 값 (예: RSI = 37.3)
- 완벽한 백테스트 결과
- 검증 구간에서 큰 성과 하락

**예방 기법**:
- 꼭 필요한 파라미터만 사용
- 가능하면 둥근 숫자 사용
- out-of-sample 테스트 필수
- 파라미터 민감도 분석 수행

### 표본 선택 Bias

**문제**: 손으로 고른 사례만 테스트(예: 알려진 시장 리더).

**핵심 문제**: 기준을 충족했지만 실패한 종목을 제외하면 전략 품질을 잘못 인식한다.

**해결**: 성공 사례만이 아니라, 기준을 충족한 **모든** 과거 사례를 테스트한다.

### Hindsight Bias

**문제**: 결과를 알고 난 뒤 그 지식이 의사결정에 영향을 줌.

**체계적 트레이딩에서의 예방**:
- 모든 규칙을 사전에 정의
- hindsight 기반 수동 개입 금지
- cherry-picking 없이 모든 사례에 규칙 테스트

### Data Mining Bias

**문제**: 무작위로 "작동"하는 전략이 나올 때까지 수백 개 전략을 테스트.

**위험**: 충분히 많이 시도하면 무작위 데이터에서도 겉보기 수익 패턴이 나온다.

**완화 방법**:
- 테스트 전에 가설 수립
- 엣지에 대한 경제적 논리 요구
- 다중 비교에 Bonferroni correction 적용
- 더 높은 유의수준 요구(p < 0.05 대신 p < 0.01)
